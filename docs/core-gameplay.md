# Capybanana 核心玩法草案

## 1. 产品定位

Capybanana 是一个竖屏 Web 随心陪伴养成小游戏。核心体验是：

> 用户每天花一分钟给宠物准备包裹和心愿，AI Agent 替宠物决定今天怎么过，宠物把这一天变成一段可收藏的成长记录。

游戏不是重度数值养成，也不是传统任务系统。用户的主要参与是“打包”和“留言”；Agent 的主要参与是“理解状态并做决定”；宠物的成长来自每天一次轻量事件。

## 2. 登录与 Agent 绑定

用户必须使用 Google 登录。

登录后，系统为用户创建或获取一个云端账号，并为该账号生成一个 Agent bind token。

用户需要把页面生成的绑定指令发给自己的 AI Agent：

```text
Read https://<app-domain>/agent/skill.md?bind=<token>
```

Agent 读取 `skill.md` 后，会获得：

- 当前宠物的基础说明
- bind token 的使用方式
- 可调用的 API 列表
- 每日玩法规则
- 当前推荐的决策流程

bind token 是 Agent 操作用户宠物的唯一凭证。用户和 Agent 操作的是同一只云端宠物。

## 3. 宠物创建与归属

每个登录用户拥有一只唯一宠物。

首次登录时，如果用户还没有宠物，系统自动创建一只默认宠物。后续用户每次登录都会读取同一只宠物的云端存档。

宠物包含：

- 基础身份
- 形象和配饰
- 精简状态参数
- 少量成长标签
- 明信片和对战记录等历史内容

## 4. 每日核心循环

每天的核心循环分为四步：

1. 用户上线
2. 用户给宠物打包
3. Agent 查看包裹和状态后做决定
4. 系统结算当天结果

宠物不会因为用户完成打包就自动行动。打包后，宠物进入 `ready` 状态，等待 Agent 调接口决定今天怎么过。

每天最多进行一次主要行动。主要行动包括：

- 旅行
- 对战
- 在家休息

当天行动完成后，用户仍可以查看记录、翻明信片、看状态，但不能再次触发主要行动。

## 5. 打包玩法

打包是用户每天最主要的互动。

用户每天最多打包 3 个物体，并可以写一小段留言。

### 5.1 物体来源

当前打包以拍照为主。

用户拍照后：

- 图片只保存在本地浏览器
- 图片不上传到云端作为存档
- 前端可以调用识别 API 判断“这是什么物体”
- 云端只保存识别后的物体信息

云端保存的数据示例：

```json
{
  "label": "水杯",
  "keyword": "水杯",
  "tags": ["warm"]
}
```

### 5.2 留言

用户可以写一小段话留给 Agent 查看。

例如：

```text
今天想去有风的地方看看。
```

Agent 在做决定前，需要读取这段留言，并结合宠物状态判断今天适合旅行、对战还是在家。

### 5.3 打包限制

- 每天最多打包 3 个物体
- 图片不做云端持久化
- 云端只保存识别结果、标签和留言
- 打包只是准备当天素材，不会自动触发出门

## 6. Agent 决策

Agent 是宠物每天行动的决策者。

Agent 每天应先调用接口查看：

- 宠物当前状态
- 今日是否已经行动
- 今日包裹内容
- 用户留言
- 宠物是否受伤
- 当前可选行动

然后 Agent 调用统一决策接口，选择今天怎么过。

推荐的决策逻辑：

- 宠物活力高、心情好、留言偏向远方：选择旅行
- 宠物勇气高、活力足、包裹看起来适合冒险：选择对战
- 宠物受伤或活力低：选择在家休息
- 用户留言明确希望慢一点、休息、在家：选择在家

Agent 不应该只机械执行用户留言，而应该像宠物的“小心思”一样，结合状态做出合理选择。

## 7. 三种主要行动

### 7.1 旅行

Agent 选择旅行后，宠物会出门一段时间。

旅行可能产生：

- 状态变化
- 旅行故事
- 有概率获得明信片
- 有概率带回纪念品或小发现

明信片是旅行的主要收藏物。

明信片内容应包含：

- 去了哪里
- 为什么去了这里
- 发生了什么
- 和用户包裹或留言的关系

### 7.2 对战

Agent 选择对战后，宠物会进行一次对战事件。

对战可能产生：

- 胜利
- 失败
- 平局
- 伤势变化
- 心情变化
- 对战记录

对战记录是另一类收藏内容，和明信片并列：

- 明信片记录旅行
- 对战记录记录战斗

对战不需要设计复杂战斗系统。MVP 中只需要根据宠物状态、少量随机性和包裹标签结算出一个结果。

### 7.3 在家休息

Agent 选择在家后，宠物当天不会出门。

在家可能产生：

- 恢复活力
- 降低伤势
- 小幅改善心情
- 一段居家故事

如果宠物受伤较重，在家休息应该是优先选择。

## 8. 宠物参数

宠物参数保持精简，避免复杂 RPG 化。

### 8.1 宠物结构

```ts
interface Pet {
  id: string
  name: string
  species: CompanionType

  appearance: {
    color: string
    accessory: Accessory
  }

  stats: {
    energy: number
    mood: number
    courage: number
    injury: number
  }

  traits: string[]
  createdAt: string
}
```

### 8.2 基础身份

```ts
id: string
name: string
species: CompanionType
createdAt: string
```

含义：

- `id`：宠物唯一 ID
- `name`：宠物名字
- `species`：宠物形象类型
- `createdAt`：领养时间

### 8.3 形象与配饰

```ts
appearance: {
  color: string
  accessory: Accessory
}
```

含义：

- `color`：主色
- `accessory`：配饰

配饰可以先保持轻量，例如：

```ts
"none" | "scarf" | "hat" | "glasses" | "flower" | "bell"
```

形象和配饰主要服务于展示，不做复杂装备系统。

### 8.4 精简状态参数

MVP 只保留 4 个核心状态值：

```ts
stats: {
  energy: number
  mood: number
  courage: number
  injury: number
}
```

#### energy 活力

范围：`0-100`

决定宠物今天是否适合出门或对战。

- 高活力：适合旅行或对战
- 低活力：适合在家休息

#### mood 心情

范围：`0-100`

影响故事语气和当天反馈。

- 高心情：故事更轻松、积极
- 低心情：更适合安静陪伴或休息

#### courage 勇气

范围：`0-100`

主要影响对战倾向和对战结果。

- 高勇气：更适合对战
- 低勇气：更适合旅行或在家

#### injury 伤势

范围：`0-100`

负面参数。

- 高伤势：不适合旅行或对战
- 低伤势：行动更自由

如果 `injury` 高于某个阈值，Agent 应优先选择在家休息。

### 8.5 成长标签 traits

```ts
traits: string[]
```

`traits` 是轻量成长标签，不是复杂技能树。

示例：

```ts
["爱晒太阳", "容易读歪留言", "喜欢收集小石头"]
```

用途：

- 展示宠物个性
- 给 Agent 判断提供参考
- 影响文案语气
- 提供长期养成感

MVP 可以限制最多 5 个 traits。

### 8.6 默认初始值

```ts
stats: {
  energy: 70,
  mood: 65,
  courage: 40,
  injury: 0
}
```

## 9. 行动对参数的影响

### 9.1 旅行

建议影响：

```ts
energy -= 10
mood += 5
courage += 1
injury += smallChance(5)
```

旅行主要消耗活力，提升心情，并带来发现明信片的机会。

### 9.2 对战

建议影响：

```ts
energy -= 15
courage += 5
mood += win ? 8 : -6
injury += lose ? random(8, 20) : random(0, 6)
```

对战主要消耗活力，提升勇气，但失败时会增加伤势。

### 9.3 在家

建议影响：

```ts
energy += 15
mood += 3
injury -= 10
```

在家主要用于恢复，是受伤或低活力时的合理选择。

## 10. 收藏与历史

MVP 至少保留两类历史记录。

### 10.1 明信片

旅行可能产出明信片。

明信片是宠物出门旅行的主要反馈，也是用户最容易理解的收藏物。

### 10.2 对战记录

对战产出对战记录。

对战记录应包含：

- 对战时间
- 胜负结果
- 简短故事
- 伤势变化
- 可选战利品

## 11. 云端数据

所有核心玩法数据都存云端 Supabase/Postgres。

包括：

- 用户账号
- Agent bind token
- 宠物基础信息
- 宠物形象
- 宠物状态参数
- 今日包裹
- 用户留言
- 旅行记录
- 明信片
- 对战记录
- traits
- 活动日志

本地只保留：

- UI 状态
- 拍照预览
- 临时图片数据

图片本身不上传云端。

## 12. MVP 范围

第一阶段基础功能：

- Google 登录
- 生成 Agent 绑定 Skill
- Agent 使用 bind token 调接口
- 每个用户唯一宠物
- 宠物包含形象、配饰和 4 个核心状态参数
- 每天最多一次打包
- 每天最多一次 Agent 决策
- 打包最多 3 个物体
- 拍照识别物体，但不上传图片
- 留言给 Agent 查看
- Agent 可选择旅行、对战或在家
- 旅行有概率产出明信片
- 对战产出对战记录
- 在家恢复状态并产生日常故事
- 核心数据云端持久化

## 13. 一句话总结

用户每天负责准备包裹和心愿，Agent 负责理解宠物状态并决定今天怎么过，宠物负责把这一天变成一段可收藏、可回看的成长记录。
