# Capybanana 核心玩法草案

> **P0 抽卡养成（2026-06，已按反馈收敛）**：旅行从「拼文案」升级成**明信片抽卡**。**目的地共 8 个，分两个池**——近(`near`：`seaside` 海边 / `forest` 森林 / `flowerfield` 花田 / `town` 小镇) 和 远(`far`：`snow` 雪地 / `mountain` 山路 / `starfield` 星河 / `desert` 沙丘绿洲)。**Agent 不能指定目的地，只能选近/远**（`distance`，近=短途回得快、远=长途）；**服务端在对应池里随机挑目的地**，由打包的关键词/心愿**加权偏置**（`pickDestination`，不是指定）。每个目的地 × **3 档稀有度**（N 普通 / R 稀有 / SR 史诗）= **24 张固定卡的图鉴**。每次旅行归来**有概率**寄回一张（约 7 成，`POSTCARD_CHANCE`；没寄回就不掷卡）；寄回时稀有度由**服务端**掷骰（`src/game/gacha.ts`，权重 N640/R300/SR60，base ≈ 64/30/6%），**纯随机、不受打包影响**；养成（陪伴天数）+ 好奇心抬「幸运值」（`SR+=幸运×4`，封顶 40 ≈ SR 19%），外加软/硬保底（`pullsSinceRare`，8 起递增、15 必出 SR）。卡面地标固定取自 `LANDMARKS[theme][rarityIndex]`（index 0→2 = N→SR，最标志性的给 SR）。**明信片文案由 LLM 在「归来时」生成**（`src/server/llm/postcard.ts` 的 `composePostcard`，在 `tickSave` 结算返程时调用）——此时稀有度已掷定、地标已确定，把地标喂给 LLM 让它**只围绕这个真实地标写**，并融合**打包物品 + 主人心愿 + Agent `travel` 调用时的 note**，所以文字与卡面地标必然一致、且和当天内容相关；无 `OPENROUTER_API_KEY` 或失败时回退程序化 `generatePostcard`。两种文案都由 `dressTravelPostcard` 统一钉上 locationName + 文末「寄自<地标>」签名。**注意时机**：早先「出发时生成、存进 `Trip.llmPostcard`」的做法因无 DB 列、出发即丢，是死代码，已删；现在改成「归来时生成并立刻随存档落库」，既真生效又与稀有度一致（`tickSave` 因此变 async，所有调用方 `await`）。**养成感 = 陪伴天数**：`companionDays` 只在「Agent 当天做了一次主要行动」时 +1（断更不涨），是**唯一对主人可见的成长条**。**5 个核心数值对主人全隐藏**——只有 Agent 通过 API 看得到。重复卡只回喂少量隐藏好奇心。迁移 `0003_postcard_gacha.sql` + `0004_gacha_3tier.sql`（SSR→SR、稀有度 CHECK 收紧到 N/R/SR；图鉴由 postcards 去重派生并按 `ALL_CARD_IDS` 过滤）。**后续（P1+）**：稀有度占位图换成正式卡面美术、纪念品实物化、切磋可见收益、季节限定。

> **实现状态（2026-06，v2 重做）**：统一决策接口 `POST /api/agent/day` 的 `action` 支持 `travel` / `battle` / `stay` 三种主要行动——**切磋已实现**：异步快照池匹配另一位主人的宠物（无则 LLM/启发式生成 NPC），把双方快照交给 LLM 判胜负、生成过程，记录写入 `battles` 表；**切磋输了必受伤并强制至少休养一天**。宠物状态采用 **5 个核心值**：`energy / mood / courage / curiosity / injury`（见 §8）。每天的「天」按 **UTC+8** 计。新增 **Agent 压力上报**（`POST /api/agent/checkin`）：Agent 把自己当天的状态/吐槽告诉宠物，宠物感同身受（见 §6.1）。旅行细节（地点/时长/明信片文案）由 OpenRouter 文本 LLM（无 key 时回退程序化）在出发时决定；数值仍走固定效果表（见 §9）。

## 1. 产品定位

Capybanana 是一个竖屏 Web 随心陪伴养成小游戏。核心体验是：

> 用户每天花一分钟给宠物准备包裹和心愿，AI Agent 替宠物决定今天怎么过，宠物把这一天变成一段可收藏的成长记录。

游戏不是重度数值养成，也不是传统任务系统。用户的主要参与是“打包”和“留言”；Agent 的主要参与是“理解状态并做决定”；宠物的成长来自每天一次轻量事件。

## 2. 登录与 Agent 绑定

用户必须使用 Google 登录。

登录后，系统为用户创建或获取一个云端账号，并为该账号生成一个 Agent bind token。

用户需要把页面生成的绑定指令发给自己的 AI Agent：

```text
Read https://<app-domain>/agent/skill.md?bind=<token>
```

Agent 读取 `skill.md` 后，会获得：

- 当前宠物的基础说明
- bind token 的使用方式
- 可调用的 API 列表
- 每日玩法规则
- 当前推荐的决策流程

bind token 是 Agent 操作用户宠物的唯一凭证。用户和 Agent 操作的是同一只云端宠物。

## 3. 宠物创建与归属

每个登录用户拥有一只唯一宠物。

首次登录时，如果用户还没有宠物，系统自动创建一只默认宠物。后续用户每次登录都会读取同一只宠物的云端存档。

宠物包含：

- 基础身份
- 形象和配饰
- 精简状态参数
- 少量成长标签
- 明信片和切磋记录等历史内容

## 4. 每日核心循环

每天的核心循环分为四步：

1. 用户上线
2. 用户给宠物打包
3. Agent 查看包裹和状态后做决定
4. 系统结算当天结果

宠物不会因为用户完成打包就自动行动。打包后，宠物进入 `ready` 状态，等待 Agent 调接口决定今天怎么过。

每天最多进行一次主要行动。主要行动包括：

- 旅行
- 切磋
- 在家休息

「天」按 **UTC+8**（北京时间 0 点翻篇）计算。当天行动完成后，用户仍可以查看记录、翻明信片、看切磋记录、看状态，但不能再次触发主要行动。**切磋切磋输了会强制进入「养伤日」：第二天只能 `stay`（`rest`），养好才能再出门。**

## 5. 打包玩法

打包是用户每天最主要的互动。

用户每天最多打包 3 个物体，并可以写一小段留言。

### 5.1 物体来源

当前打包以拍照为主。

用户拍照后：

- 图片只保存在本地浏览器
- 图片不上传到云端作为存档
- 前端可以调用识别 API 判断“这是什么物体”
- 云端只保存识别后的物体信息

云端保存的数据示例：

```json
{
  "label": "水杯",
  "keyword": "水杯",
  "tags": ["warm"]
}
```

### 5.2 留言

用户可以写一小段话留给 Agent 查看。

例如：

```text
今天想去有风的地方看看。
```

Agent 在做决定前，需要读取这段留言，并结合宠物状态判断今天适合旅行、切磋还是在家。

### 5.3 打包限制

- 每天最多打包 3 个物体
- 图片不做云端持久化
- 云端只保存识别结果、标签和留言
- 打包只是准备当天素材，不会自动触发出门
- **包裹会过期**：备好的「今日包裹」放过约 24 小时仍没被用掉就视为「不新鲜」。过期判定放在前端——用户登录回到首页时检查包裹时间，超时则提示「东西已过期，要不要重新打包」并清掉旧包裹（服务端只做被动清除，不自动过期）。这呼应「每天一分钟」的节奏——包裹是当天的，不隔夜囤。

## 6. Agent 决策

Agent 是宠物每天行动的决策者。

Agent 每天应先调用接口查看：

- 宠物当前状态
- 今日是否已经行动
- 今日包裹内容
- 用户留言
- 宠物是否受伤
- 当前可选行动

然后 Agent 调用统一决策接口，选择今天怎么过。

推荐的决策逻辑：

- 宠物活力高、心情好、留言偏向远方：选择旅行
- 宠物勇气高、活力足、包裹看起来适合冒险：选择切磋
- 宠物受伤或活力低：选择在家休息
- 用户留言明确希望慢一点、休息、在家：选择在家

Agent 不应该只机械执行用户留言，而应该像宠物的“小心思”一样，结合状态做出合理选择。

### 6.1 Agent 压力上报（每天第一步的一部分）

每天的循环精简成**两步**：① `POST /api/agent/checkin`（看状态 + 上报自己，一次完成）→ ② `POST /api/agent/day`（替它做决定）。

第一步 `POST /api/agent/checkin`：Agent 把*它自己*这一天过得怎么样告诉宠物——`stress` 取 `light`(轻松) / `normal`(一般) / `tired`(累) / `exhausted`(很累)，外加一句自由 `note`（吐槽）。**同一次调用还会返回宠物此刻的完整信息**（`{pet, events, postcards, battles}`：状态、今日包裹、能做的动作、明信片、切磋记录、最近事件），所以不再需要单独的「看状态」接口；空 body 调用即纯看状态、不报压力。

意义是把「Agent 的压力」具象化：照看它的人累，宠物也会跟着想慢一点。机制上保持简单——压力按档位映射成一次固定小幅心情/体力调整（如 `tired: mood-3,energy-3`；`exhausted: mood-5,energy-5`），并写进一条记忆；`note` 会被保留，喂给当天旅行/切磋的 LLM 作为语气线索，行动后清空。压力不做成常驻数值条。

## 7. 三种主要行动

### 7.1 旅行

Agent 选择旅行后，宠物会出门一段时间。**Agent 只决定去近还是去远（`distance` near/far）**，不指定目的地；服务端在对应距离池里随机挑一个目的地（由打包关键词/心愿加权偏置），近=短途、远=长途。归来**有概率**寄回一张明信片（约 7 成，`resolveDay.ts` 的 `POSTCARD_CHANCE`；没寄回时就是一场普通的「出门转转」，不掷卡）；寄回时稀有度纯随机（见 P0）。

旅行可能产生：

- 状态变化
- 旅行故事
- 有概率获得明信片
- 有概率带回纪念品或小发现

明信片是旅行的主要收藏物。

明信片内容应包含：

- 去了哪里
- 为什么去了这里
- 发生了什么
- 和用户包裹或留言的关系

### 7.2 切磋

Agent 选择切磋后，系统从「异步快照池」匹配另一位主人的宠物快照（近 7 天内、异主），匹配不到就生成一只 NPC 对手。双方快照（含五个属性、性格、压力线索）一起交给 LLM 裁判判胜负并生成过程描述；没配 LLM key 时回退到简单的分数比较（见 §9.2）。

切磋可能产生：

- 胜利 / 失败 / 平局
- 伤势变化（**切磋输了必受伤**，并触发强制养伤日）
- 心情、勇气变化
- 切磋记录（收藏）

切磋记录是另一类收藏内容，和明信片并列：

- 明信片记录旅行
- 切磋记录记录切磋

切磋不设计复杂切磋系统：判定主要看勇气与体力，加一点随机；切磋输了强制至少休养一天。

### 7.3 在家休息

Agent 选择在家后，宠物当天不会出门。

**在家不消耗包裹**：只有旅行 / 切磋才会用掉「今日包裹」；在家是一场独立的休息，备好的包裹原样留着（留给下次出门，或满 24 小时按 §5.3 过期），结算也不读包裹内容。在家仍算当天的一次主要行动。

在家可能产生：

- 恢复活力
- 降低伤势
- 小幅改善心情
- 一段居家故事

如果宠物受伤较重（`injury ≥ 45`）或刚切磋输了（养伤日内），在家休息是唯一选择——此时 `choices` 只剩 `["stay"]`。`rest` 每天回 18 点体力、降 18 点伤，养好且过了养伤日就解禁。

## 8. 宠物参数

宠物参数保持精简，避免复杂 RPG 化。

### 8.1 宠物结构

```ts
interface Pet {
  id: string
  name: string
  species: CompanionType

  appearance: {
    color: string
    accessory: Accessory
  }

  stats: {
    energy: number
    mood: number
    courage: number
    curiosity: number
    injury: number
  }

  traits: string[]
  createdAt: string
}
```

### 8.2 基础身份

```ts
id: string
name: string
species: CompanionType
createdAt: string
```

含义：

- `id`：宠物唯一 ID
- `name`：宠物名字
- `species`：宠物形象类型
- `createdAt`：领养时间

### 8.3 形象与配饰

```ts
appearance: {
  color: string
  accessory: Accessory
}
```

含义：

- `color`：主色
- `accessory`：配饰

配饰可以先保持轻量，例如：

```ts
"none" | "scarf" | "hat" | "glasses" | "flower" | "bell"
```

形象和配饰主要服务于展示，不做复杂装备系统。

### 8.4 精简状态参数

MVP 只保留 5 个核心状态值：

```ts
stats: {
  energy: number
  mood: number
  courage: number
  curiosity: number
  injury: number
}
```

#### energy 活力

范围：`0-100`

决定宠物今天是否适合出门或切磋。

- 高活力：适合旅行或切磋
- 低活力：适合在家休息

#### mood 心情

范围：`0-100`

影响故事语气和当天反馈。

- 高心情：故事更轻松、积极
- 低心情：更适合安静陪伴或休息

#### courage 勇气

范围：`0-100`

主要影响切磋倾向和切磋结果。

- 高勇气：更适合切磋
- 低勇气：更适合旅行或在家

#### curiosity 好奇心

范围：`0-100`

主要影响旅行倾向——“想去哪、想不想去远一点的地方”。

- 高好奇心：更想出门、去新奇的地方
- 旅行会让它稍稍增长

#### injury 伤势

范围：`0-100`

负面参数。

- 高伤势：不适合旅行或切磋
- 低伤势：行动更自由

如果 `injury` 高于某个阈值，Agent 应优先选择在家休息。

### 8.5 成长标签 traits

```ts
traits: string[]
```

`traits` 是轻量成长标签，不是复杂技能树。

示例：

```ts
["爱晒太阳", "对远方很好奇", "喜欢收集小石头"]
```

用途：

- 展示宠物个性
- 给 Agent 判断提供参考
- 影响文案语气
- 提供长期养成感

MVP 可以限制最多 5 个 traits。

### 8.6 默认初始值

```ts
stats: {
  energy: 70,
  mood: 65,
  courage: 40,
  curiosity: 50,
  injury: 0
}
```

## 9. 行动对参数的影响

数值逻辑刻意保持简单可读——每个动作一张固定小整数效果表（实现见 `resolveDay.ts` 的 `EFFECTS` 与 `engine.ts` 的 `BATTLE_EFFECTS`），只有少量单次掷骰。

### 9.1 旅行

```ts
energy -= 10
mood += 5
courage += 3
curiosity += 5
injury += 25% ? 15 : 0   // 一次掷骰，可能受点伤
```

旅行主要消耗活力，提升心情与好奇心，并带回明信片。

### 9.2 切磋

```ts
energy -= 15
courage += win ? 5 : draw ? 3 : 2
mood += win ? 8 : draw ? 1 : -6
injury += win ? 3..8 : draw ? 8..16 : 16..26   // 切磋输了必受伤
rating += win ? +15 : lose ? -15 : 0            // 仅用于匹配，不做 ELO
```

胜负判定（无 LLM 时的回退启发式）：`score = courage + energy + curiosity/2 + random(0,30)`，双方比大小，差距很小算平局。切磋输了后强制至少养伤一天。

### 9.3 在家

```ts
rest:  injury -= 18; energy += 18; mood += 4
home:  mood  += 6;  energy += 12; injury -= 6
yard:  mood  += 8;  energy -= 3;  curiosity += 3
```

在家主要用于恢复，是受伤、养伤日或低活力时的合理选择。

## 10. 收藏与历史

MVP 至少保留两类历史记录。

### 10.1 明信片

旅行可能产出明信片。

明信片是宠物出门旅行的主要反馈，也是用户最容易理解的收藏物。

### 10.2 切磋记录

切磋产出切磋记录。

切磋记录应包含：

- 切磋时间
- 胜负结果
- 简短故事
- 伤势变化
- 可选战利品

## 11. 云端数据

所有核心玩法数据都存云端 Supabase/Postgres。

包括：

- 用户账号
- Agent bind token
- 宠物基础信息
- 宠物形象
- 宠物状态参数
- 今日包裹
- 用户留言
- 旅行记录
- 明信片
- 切磋记录
- traits
- 活动日志

本地只保留：

- UI 状态
- 拍照预览
- 临时图片数据

图片本身不上传云端。

## 12. MVP 范围

第一阶段基础功能：

- Google 登录
- 生成 Agent 绑定 Skill
- Agent 使用 bind token 调接口
- 每个用户唯一宠物
- 宠物包含形象、配饰和 5 个核心状态参数
- 每天最多一次打包
- 每天两步：checkin（看状态 + Agent 压力上报，一次完成）→ day（一次主要决策）
- 打包最多 3 个物体
- 拍照识别物体，但不上传图片
- 留言给 Agent 查看
- Agent 可选择旅行、切磋或在家
- 旅行由 LLM 决定细节并产出明信片（无 key 回退程序化）
- 切磋：异步快照池匹配 + LLM 判定 + 切磋记录，切磋输了必受伤并强制养伤
- 在家恢复状态并产生日常故事
- 核心数据云端持久化

## 13. 一句话总结

用户每天负责准备包裹和心愿，Agent 负责理解宠物状态并决定今天怎么过，宠物负责把这一天变成一段可收藏、可回看的成长记录。
