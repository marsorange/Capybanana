# Capybanana — project orientation

竖屏 Web 随心陪伴养成小游戏。核心气质是“每天一分钟，陪低边形卡皮巴拉长大一点”：短互动、轻养成、卡片收藏、移动端优先。MVP 仍保持全程序化低多边形 3D，无外部美术素材、无 AI/图片 API。

- **状态/路由**：`src/state/gameStore.ts`（Zustand + persist，`skipHydration`，客户端 `rehydrate()`）。屏幕用 `screen` 状态机切换，不用路由。`GameRoot.tsx` 每秒 `tick()` 推进生命周期 + 窗口聚焦补算。
- **生命周期**：`idle_home → ready(行李已备) → traveling → idle_home`。纯逻辑在 `src/game/clock.ts`（`advanceLifecycle` 一次循环结算出门+旅行+归来，支持长时间离线补算；低概率“今天不出门”会自恢复）。
- **内容/规则**：`src/game/destinations.ts`（10 主题文案 + 关键词规则）、`planTrip.ts`（加权随机选目的地，对玩家隐藏）、`generatePostcard.ts`（按留言/行李/性格拼装文案）。
- **3D（home 与 主角已拆分）**：`src/components/scenes3d/` 分三层——`home/`（环境：`Island`/`House`/`Yard`/`villaLayout`/`Backpack`/`HomeScene`）、`character/`（主角：见下）、根目录放共享件（`SceneCanvas`/`materials`/`commandBus`/`zoomBus`/`InteractionLayer`/`InteractionMarker`/`RoamingCompanion` 这个把主角放进 home 场景的胶水）。统一 `MeshToonMaterial + gradientMap` + drei `Outlines`；不要给 `meshToonMaterial` 传 `flatShading` prop（R3F 类型不支持，会编译失败）。
- **主角形象（6 个固定角色）**：物种就是 `CompanionType` = `capybara|rabbit|duck|raccoon|shiba|sheep`，花名册唯一真源在 `src/game/characters.ts`（`CHARACTERS`：每个角色的中文名/默认色/招牌配饰/耳朵比例/参考图名 + `normalizeSpecies` 旧存档兜底映射）。参考美术图在 `src/asset/Character/*.png`（Capybanana/Bunberry/Quackaroo/Raccoonie/Shibuddy/Woollybean），**专属 3D 资产后续再生成**——当前 6 个物种都渲染共享占位体 `character/CreatureModel.tsx`（按 type/color/accessory 参数化，按花名册耳朵比例区分；眼睛偏大、加腮红走可爱路线）。渲染统一走 `character/Character3D.tsx`（公开入口，按物种查 `character/models.ts` 注册表 → 模型组件，目前全指向 `CreatureModel`；某物种真 3D 就位时只换 `models.ts` 那一行）。`labels.ts` 的 `COMPANION_TYPES`/`TYPE_LABELS`、`postcardPrompt` 的物种描述都从花名册派生。**换形象**：新账号默认领养随机一个角色（`randomCuteCompanion`：现在就是从 6 角色里随机挑一个，连默认色+招牌配饰一起套，读起来就是那只参考角色）；已有宠物改外观走 `engine.restyleCompanion`（只改 type/color/accessory，不动名字/数值/历史），入口三个——`POST /api/agent/restyle`（`{random}` 或指定字段）、ProfileScreen 的「🎲 换个样子」按钮、测试链接 `GET /agent/restyle?bind=<token>`。`coerce*` 现接受任意 `#rrggbb` 颜色（不再限定调色板）。
- **家场景**：等距 diorama，`home/HomeScene.tsx` = `Island`(浮空草地小岛) + `House`(两层剖面别墅，向 +x/+z 开切口) + `Yard`(院子：栅栏/踏石/树/菜地/花)。镜头是**正交等距 + 双指缩放**（`SceneCanvas` 的 `orthographic`/`enableZoom`/`azimuth`/`minZoom`..`maxZoom`）。主角 `RoamingCompanion.tsx` 在 `home/villaLayout.ts` 的 SPOTS（含院子）间寻路走动+活动，仅在家时出现。
- **打包（已极简）**：`PackScreen` 只有一个放大的 3D 背包 + 📷拍照 + 💬留言 + 确认，无预设网格、无快捷标签。可拍多张，每拍一张确认后自动进背包（`photos: PackedItem[]` 缩略图列，可单独移除）。每样 `PackedItem`（`preset` 仅 Agent API 用 / `photo`）。`CameraCapture.tsx` 拍完先用 `photoExtract.ts` 取主色得“氛围”标签（`tags` 仍按颜色，喂 outcome），再 `POST /api/recognize` 调 MiniMax 视觉理解（`src/lib/minimax.ts` 的 `understandImage`→Token Plan 视觉端点 `/v1/coding_plan/vlm`，复用 `MINIMAX_API_KEY`）得“这是什么”，把物体名填进 `hint`/`keyword`/`label`；没配 key 或识别失败则保留取色启发式，不打断。摄像头需安全上下文（localhost/https）。
- **账号 / Agent 连接（云存档）**：**只用 Supabase Auth 登录（Google）**，已移除手机号登录与访客「本地玩」入口——未登录（`!cloud`）只能看到登录页。流程是 **bridge 模式**：浏览器用 `@supabase/supabase-js`（`src/lib/supabaseClient.ts` 的 `signInWithGoogle`）走 Google OAuth，登录后 `GameRoot` 的 `useSupabaseAuthBridge` 监听到 Supabase session，把 `access_token` POST 给 `POST /api/auth/supabase`；服务端 `supabase.auth.getUser(token)` 验证身份（对 Supabase 任意登录方式都通用），按 `sb:<supabaseUserId>` 在 `store.loginBySupabase` 找/建账号，铸出我们自己的 `bindToken`（`/api/agent/*` 的唯一凭证，Agent 也用它；Supabase 只认证主人，不碰 bindToken 会话）。需要两个公开环境变量 `NEXT_PUBLIC_SUPABASE_URL` / `NEXT_PUBLIC_SUPABASE_ANON_KEY`（未配置则登录页显示「登录暂不可用」、`/api/auth/supabase` 回 503）；Supabase 后台要开 Google provider 并把站点/回调 URL 加进 URL Configuration。**登录即可直接进游戏**：supabase route 发现账号没宠物时直接在服务端 `createPet(save, randomCuteCompanion(), now)` 随机领养一只，返回的 save 已带 `companion`，**不再卡在连接页等 Agent 建宠**（这曾是“登录后进不去游戏”的 bug）。`logout()` 会一并 `supabase.auth.signOut()`，否则 bridge 会立刻把你重新登回去。登录/领养后先落到 `ProfileScreen`（`screen: "profile"`，旋转 3D 形象 + 性格/外观 chips + 当前参数条 + 习得性格 + 领养日期），点「进入小屋 →」才 `goTo("home")`；`home` 顶部名字块可点回看 `profile`。客户端兜底 `gameStore.ensureCloudPet`（`GameRoot` 在 `bound && !companion` 时调一次、期间显示 Splash）救活历史上被留成无宠的旧账号。连接 AI Agent 变成可选：从 `home` 右上「Agent」按钮进 `ConnectAgentScreen` 拿绑定链接 `Read <域名>/agent/skill.md?bind=<token>`（页面只展示这一句「绑定指令」，令牌已内嵌其中，不再单独列「绑定码」框）。AI Agent 读 `skill.md`（`src/app/agent/skill.md/route.ts`，按 token 动态生成带真实接口的说明）后，用同一套 **`/api/agent/*`**（`pet/create/pack/travel/battle/stay/pat/say/diary/restyle/collect/postcards/feed`，bind 令牌鉴权：`?bind=` 或 `Authorization: Bearer`）操作**同一只**云宠物。服务端复用纯逻辑：`src/server/engine.ts` 的 `tickSave` 包 `advanceLifecycle`，outcome 合并抽到共享的 `src/game/applyOutcome.ts`（`gameStore.tick` 也用它）；存储在 `src/lib/kv.ts`（Upstash/Vercel KV，未配置则进程内存回退）+ `src/server/store.ts`。登录后网页变“瘦客户端”：`gameStore` 的 `cloud` 切片把 `prepareBag/collectPostcard/createCompanion` 路由到服务器，`GameRoot` 改用 `cloudPull` 每 5s 拉取、本地不再随机结算。
- **Agent 驱动的玩法（核心）**：云宠物**不再自己出门**——主人在网页备好「今日包裹」（物品 + 心愿），宠物停在 `ready` 待命，由 **Agent** 看状态 + 包裹（`summarizePet` 现含 `bag`/`choices`）后替它决定今天怎么过：`startTravel`（旅行，可指定 `destination`，归来寄明信片）/ `startBattle`（找 Claw 较量，`resolveDay` 的 `resolveClaw` 按勇敢+体力+`protective`物+羁绊+运气出胜/负/平+战利品；`DayOutcome.battle` 标记胜负，`applyOutcome.battleRecordFrom` 把它蒸成 `CloudSave.battles` 的一条**对战记录**——明信片=旅行记录，对战记录=打架记录，两边都在 `AlbumScreen` 的标签页里展示，客户端 `tick`/`devRunDay` 同样落本地 `battles`）/ `stayHome`（在家/院子/养伤，立即结算）。机制：`Trip.intent`（`OutcomeKind|"auto"|"quiet"`）决定 `resolveDay` 走哪种收场；`engine.packBag` 用 `clock.ts` 的 `NO_AUTO_DEPART` 哨兵把 `departAt` 设成永不触发，所以 `advanceLifecycle` 的 ready→traveling 自动出门只在**本地访客模式**（`gameStore.prepareBag` 仍 `scheduleDeparture`）触发，云端纯靠 Agent。**日记**：Agent 每天 `POST /api/agent/diary`（`{text}` ≤200 字，以宠物口吻）写一篇，`engine.writeDiary` 按 UTC 日期 upsert（同日覆盖、留近 90 篇）存 `CloudSave.diary`，`summarizePet` 暴露 `latestDiary`/`wroteDiaryToday`，`GET /api/agent/diary?limit=` 回看。日记还有个**藏在背面的 `gripe`**（可选）：跳出宠物甜口吻、用「被使唤的 Agent」语气吐槽「今天又干了很多活」式的牢骚，主人在 `DiaryPanel` 点卡片翻面才看到；Agent 不传则 `game/diaryGripes.ts` 的 `pickGripe()` 兜底。`DiaryPanel` 现在读真实云端 `diary`（`adoptSave` 已带过来），访客无日记时退回「把明信片当日记」的占位。**已知缺口**：`secretProgress`（多日秘密铺垫）与明信片生图持久化仍是客户端本地的，云宠物暂不享有，后续可下沉到服务端。
- **部署（Vercel）**：所有 route 文件均 `runtime="nodejs"`+`dynamic="force-dynamic"`。**云存档需要一个持久化后端**（否则 Vercel 多实例下进程内存兜底会丢令牌→`/api/agent/*` 401，登录后被踢回登录页）。`kv.ts` 三选一自动识别：①Redis（Upstash/Vercel KV：`UPSTASH_REDIS_REST_URL/TOKEN` 或 `KV_REST_API_URL/TOKEN`）；②Postgres（如 Supabase：`POSTGRES_URL`/`POSTGRES_PRISMA_URL`/`POSTGRES_URL_NON_POOLING`，用 `postgres` 库连 pgbouncer 池，`prepare:false`，首用自动建 `capy_kv`/`capy_kv_set` 两张表，无需手动 SQL）；③都没有则进程内存（仅本地单进程可用）。`KV_PERSISTENT`/`KV_BACKEND` 暴露当前后端；login route 在 `process.env.VERCEL && !KV_PERSISTENT` 时直接回 503 提示去配数据库，而不是给一个注定 401 的会话。`/agent/skill.md` 用点号文件夹路由，已验证 `next build` 可注册、无需 rewrite。绑定链接域名从请求头推导，可用 `APP_BASE_URL` 固定。
- **UI 主题**：Tailwind v4 `@theme` in `globals.css`（`bg-cream`/`text-ink`/`bg-accent #D95F59`/`rounded-sticker`/`font-hand`）。标题字体用系统 CJK（Yuanti/PingFang），不外链 Google Fonts。`ErrorBoundary.tsx` 包住每屏，3D 出错只显示重试卡片不至于全白。
- **验证 & 坑**：dev 运行时用 `npx tsc --noEmit`，**不要在 `next dev` 运行时跑 `next build`**（两者共用 `.next`，会让 dev 加载不稳定/白屏；修复：`rm -rf .next` 后重启 dev）。
- 仓库目录名 `Capybanana` 含大写，npm 包名不能含大写——脚手架曾用小写临时子目录再上移。
- **远期玩法蓝图（用户已提出，尚未实现）**：纪念品/特产、负伤与休养、成长、对战与战后恢复、家庭活动、按“天”推进的日循环；在家自主生活由 时间/心情/体力/是否受伤/纪念品 驱动选择活动。当前只实现到“出门旅行→寄明信片→带回”闭环。
