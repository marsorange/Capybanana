# Capybanana — project orientation

竖屏 Web 随心陪伴养成小游戏。核心气质是“每天一分钟，陪低边形卡皮巴拉长大一点”：短互动、轻养成、卡片收藏、移动端优先。主体验仍保持低多边形 3D；GLB 管线暂时保留但默认关闭，明信片 AI 生图仍下线。

- **状态/路由**：`src/state/gameStore.ts`（Zustand + persist，`skipHydration`，客户端 `rehydrate()`）。屏幕用 `screen` 状态机切换，不用路由。登录后客户端只投影云端 `CloudSave`，不再保留访客/本地自动结算玩法；`GameRoot.tsx` 通过 `cloudPull()` 轮询服务端状态。
- **生命周期**：`idle_home → ready(行李已备) → traveling → idle_home`。纯逻辑在 `src/game/clock.ts`；`ready` 不会自动出门，必须由 Agent 调 `/api/agent/day` 决定。`advanceLifecycle` 只负责把已经 `traveling` 的云端行程在到时后结算归来。
- **内容/规则**：`src/game/destinations.ts`（**8 个目的地，分 near/far 两池** + 关键词规则 + 每地 3 档稀有度地标 `LANDMARKS`）、`planTrip.ts`（`pickDestination`：在 Agent 选定的 near/far 池里、按打包关键词**加权随机**选目的地，对玩家隐藏）、`gacha.ts`（3 档稀有度 N/R/SR、24 张图鉴、服务端掷骰，**稀有度纯随机、不受打包影响**）、`generatePostcard.ts`（程序化拼装文案，作为 LLM 无 key 时的回退）。**明信片文案在归来时由 LLM 写**：`server/llm/postcard.ts` 的 `composePostcard` 拿到已定死的稀有度+地标，融合打包物品/留言/Agent note 写文案；`engine.dressTravelPostcard` 统一钉上地标 + 文末「寄自」签名（卡面与文字必一致）。
- **3D（只有两个模型：home + 主角）**：`src/components/scenes3d/` 故意收敛成「两个模型 + 舞台/系统」。**对外只有两个模型入口**：`home/HomeModel.tsx`（整个家场景）和 `character/CharacterModel.tsx`（主角）。其余都是支撑层，不是「模型资产」：根目录 `SceneCanvas`(相机/灯光舞台)、`materials`(共享 toon 材质)、`zoomBus`、`RoamingCompanion.tsx`(把主角放进 home 场景 + 寻路的胶水)。统一走 `materials.ts` 的 **`toonMaterial(color, opts?)`** 工厂（缓存版 `MeshToonMaterial` + gradientMap cel ramp + Fresnel 边缘光 + 注入式「星球弯曲」顶点位移）+ drei `Outlines`；各 part 文件原来的 `m()`/`glow()` 助手现在都委托成 `<primitive object={toonMaterial(c)} attach="material" />`（共享实例，N 个同色网格共用一个已编译程序）。**坑**：别在 JSX 里给 `<meshToonMaterial>` 传 `flatShading`（R3F 类型不支持，会编译失败）——工厂在 TS 里 `new MeshToonMaterial(...)` 后用窄 cast 设 `flatShading`，绕开它。「星球弯曲」是 `sceneBend` 共享 uniform，由 `SceneCanvas` 的 `bendStrength` 开关（home 场景给 0.15、转盘屏默认 0=不弯）：**只弯视觉网格、碰撞体不弯**，宠物走的中心区 bend≈0 所以两边不打架。**加新模型/换 3D 时只改这两个入口，别再往外散文件。**
- **主角模型（`character/CharacterModel.tsx`，单文件）**：一个组件搞定——按 `type` 解析后统一归到当前单主角卡皮巴拉（已内含 `normalizeSpecies`，旧类型也稳）。**没有 dispatcher/registry**（之前的 `Character3D`/`models.ts` 已删）。`CompanionType` 仍保留 `capybara|rabbit|duck|raccoon|shiba|sheep` 只是为了旧存档/Agent 输入兼容；花名册唯一真源在 `src/game/characters.ts`，当前 `CHARACTERS` 只有 `Capybanana`，参考图只保留 `src/asset/Character/Capybanana.png`。**GLB 资产管线先保留**：`character/companionModels.ts` 仍是「物种 → GLB」清单（数据，不是文件 registry），但 `GLB_PIPELINE_ENABLED=false`，所以当前主角走程序化体；运行时 GLB 仍放 `public/models/`，Draco 解码器仍在 `public/draco/`。之后修好资产只需打开开关并在清单加回 `capybara: "/models/Capybanana.glb"`，调用方不用改。`labels.ts` 的 `COMPANION_TYPES`/`TYPE_LABELS` 从花名册派生。**换形象**：新账号**不再自动领养**，宠物由 Agent 调 `POST /api/agent/create` 绑定并注册（含取名，详见账号连接一节）；已有宠物改外观走 `engine.restyleCompanion`（只改 type/color/accessory，不动名字/数值/历史），入口三个——`POST /api/agent/restyle`、ProfileScreen 的「🎲 换个样子」按钮、测试链接 `GET /agent/restyle?bind=<token>`。`coerce*` 现接受任意 `#rrggbb` 颜色。
- **家模型（`home/HomeModel.tsx`，单入口）**：等距 diorama = `Island`(浮空草地小岛) + `House`(两层剖面别墅，向 +x/+z 开切口) + `Yard`(院子：栅栏/踏石/树/菜地/花)。这三块几何是**私有部件**，放在 `home/parts/`（含 `Backpack.tsx`，PackScreen 也复用它），外部一律只 import `HomeModel`。配套：`home/layout.ts`(原 `villaLayout`，SPOTS/楼梯等寻路数据)、`home/interaction/`(`InteractionLayer`/`InteractionMarker`/`commandBus`/`navBus`)。镜头是**正交等距 + 双指缩放**（`SceneCanvas` 的 `orthographic`/`enableZoom`/`azimuth`/`minZoom`..`maxZoom`）。**注意：当前 home 美术是占位**，会整体重做——别在这堆装饰几何上花功夫优化（instancing 等），等新美术定稿。
- **物理框架（@react-three/rapier = Rapier 的 Rust→WASM 物理）**：目标是「整页真实物理手感」。`SceneCanvas` 加了 `physics` 开关（包一层 `<Physics gravity debug>`，只有 home 场景开；人像转盘屏不开）。**核心铁律：渲染网格 ≠ 碰撞体**——HomeModel 那几百个装饰网格全是纯视觉，物理世界只认 `home/HomeColliders.tsx` 里那十来个隐形 fixed collider（地面/外墙/两面实墙，尺寸照 `layout.ts` 近似；美术重做时换视觉、碰撞体契约不变）。HomeColliders 的隐形地面 plane 还兼「点地走」拾取器（写 `navBus`）。主角 `RoamingCompanion.tsx` 已是 **Rapier `KinematicCharacterController`**：kinematic 刚体 + 胶囊 collider，每帧「朝目标的水平位移 + 重力」交给 controller 解算（贴地、爬小台阶、贴墙滑行不穿模）。**无用户操控**，目标只来自三种——点地(`navBus`) / 场景 marker(`commandBus`，含日记入口回调) / 自动在 floor-0 的 SPOTS 间漫游；楼梯/上下楼暂缓（等美术加真坡道）。物理道具样板 `home/PhysicsToy.tsx`（dynamic `<RigidBody>` 小球，点一下抛出会弹会滚）——以后背包物品掉落、积木堆叠都照这个抄。新 3D：**静态装饰**用 `materials.ts` 的 `toonMaterial(color)`（cel + 弯曲 + 边缘光）；**动态物理道具**用 `flatMaterial(color)`（缓存版 Lambert，刻意不进 toon/不进星球弯曲——滚动的球不该被掰弯，且省一个 shader 给物理腾余量）。坑：磁盘上有两份 `@dimforge/rapier3d-compat`（顶层 0.12 仅 `@types/three` 类型用，运行时走 @react-three/rapier 的 0.19），所以别裸 `import` 它的类型，要从 `world.createCharacterController` 的返回类型推。
- **打包（已极简）**：`PackScreen` 内联相机取景 + 📷拍照 + 💬留言 + 确认，无预设网格、无快捷标签。可拍多张，每拍一张确认后自动进背包（`photos: PackedItem[]` 缩略图列，可单独移除）。每样 `PackedItem`（`preset` 仅 Agent API 用 / `photo`）。拍完先用 `photoExtract.ts` 取主色得“氛围”标签（`tags` 仍按颜色，喂 outcome），再 `POST /api/recognize` 调 MiniMax 视觉理解（`src/lib/minimax.ts` 的 `understandImage`→Token Plan 视觉端点 `/v1/coding_plan/vlm`，复用 `MINIMAX_API_KEY`）得“这是什么”，把物体名填进 `hint`/`keyword`/`label`；没配 key 或识别失败则保留取色启发式，不打断。摄像头需安全上下文（localhost/https）。**包裹会过期（前端判定）**：备好的「今日包裹」放过约 **24 小时**没被用掉就算「不新鲜」。**过期判定在前端、不在服务端生命周期**——网页进首页时 `HomeScreen` 调 `gameStore.expireStaleBag()` 比对 `packedBag.packedAt`（阈值 `BAG_TTL_MS`，可用 `NEXT_PUBLIC_CAPY_BAG_TTL_MIN` 在 dev 压到几分钟）；超时则立刻弹 `notice` 提示（瞬态、不持久化，`HomeScreen` 的 `NoticeToast` 渲染「东西不新鲜，要不要重新打包」），并调 `POST /api/agent/unpack` 让服务端做一次**哑清除**（`engine.clearBag`：清 bag + 宠物回 `idle_home` + bump `bagExpired` 事件，落库 `activities.kind='pack'`、真类型在 `payload.event_type`，无需迁移）。`tickSave`/`advanceLifecycle` **不再自动过期包裹**，服务端只在收到 unpack 请求时清。
- **账号 / Agent 连接（云存档）**：**只用 Supabase Auth 登录（Google）**，已移除手机号登录与访客「本地玩」入口——未登录（`!cloud`）只能看到登录页。流程是 **bridge 模式**：浏览器用 `@supabase/supabase-js`（`src/lib/supabaseClient.ts` 的 `signInWithGoogle`）走 Google OAuth，登录后 `GameRoot` 的 `useSupabaseAuthBridge` 监听到 Supabase session，把 `access_token` POST 给 `POST /api/auth/supabase`；服务端 `supabase.auth.getUser(token)` 验证身份，按 Supabase user id 在 `store.loginBySupabase` 找/建账号，铸出我们自己的 `bindToken`。**登录不再自动建宠物**——账号默认 petless，`GameRoot` 把人停在「接入 Agent」门槛页（`ConnectAgentScreen`）轮询等待，直到 Agent 拿 bindToken 调 `POST /api/agent/create` 绑定并注册宠物（**名字也由 Agent 取**）后才放行进岛（`未绑定无法计入岛屿`）。`bindToken` 是凭证、`/api/agent/create` 是绑定动作：网页端与 Agent 共用同一 token，`resolveBind` 对任何调用都会刷 `last_used_at`，所以「Agent 是否真的接入」唯一可靠信号就是宠物是否被创建（`savePet` 本就拒绝 petless 存档，宠物=唯一可持久化的绑定凭据）。云存档 **SQL-only**，只认 `POSTGRES_URL`/`POSTGRES_PRISMA_URL`/`POSTGRES_URL_NON_POOLING`，需依次执行 `supabase/migrations/0001_storage_refactor.sql`、`supabase/migrations/0002_gameplay_v2.sql`；没有 PostgreSQL 时登录 route 直接 503。AI Agent 读 `skill.md` 后用 `/api/agent/*` 操作同一只云宠物；网页端只是持有同一个 bind token 的瘦客户端。
- **宠物数值 / 玩法真源**：`docs/core-gameplay.md` 是核心玩法的设计真源（init 后请优先读它对齐）。宠物状态是 **5 个核心值**——`energy / mood / courage / curiosity / injury`（`CapyState`，外加轻量 `traits` 成长标签与 `memories` 短记忆；`curiosity 好奇心`在 v2 加回，驱动旅行倾向；bravery 早已并入 courage）。`DEFAULT_CAPY` = energy 70 / mood 65 / courage 40 / curiosity 50 / injury 0。**数值刻意简单**：每个动作一张固定小整数效果表（`resolveDay.ts` 的 `EFFECTS`、`engine.ts` 的 `BATTLE_EFFECTS`），别堆复杂公式/ELO。另有 **「压力」**——不是常驻属性，是 Agent 每天 `checkin` 上报的临时值（`pendingStress`/`pendingStressNote`，被当天行动消费）。`traits` 现在真的会涨（`resolveDay` 在难忘的一天小概率授予）。改任一处数值都要同步 `types.ts`/`store.ts`/迁移 SQL/`skill.md`/Profile·Result·Album。
- **Agent 驱动的玩法（核心，每天三步）**：云宠物**不自己出门**——主人在网页备「今日包裹」，宠物停在 `ready` 待命。Agent 每天：① `GET /api/agent/pet` 看状态 → ② `POST /api/agent/checkin`（`{stress, note}`，把 Agent 自己累不累告诉宠物，宠物感同身受 + 喂给当天**对战** LLM 作语气线索）→ ③ `POST /api/agent/day` 统一决策，`action` 三选一：`travel`（**Agent 只选 `distance` near/far**，服务端在对应池里按打包加权随机定目的地、按距离定时长；归来时 `dressTravelPostcard` 掷稀有度 + 定地标 + **LLM 写明信片**（融合打包/留言/note，无 key 回退程序化），随存档落库——`tickSave` 因此为 async）/ `battle`（**已实现**：`findBattleOpponent` 从 `battle_pool` 匹配异主宠物快照，无则 `makeNpcOpponent` 生 NPC，`judgeBattle` 用 OpenRouter 文本判胜负、无 key 走启发式；写 `battles` 表 + 刷新 `battle_pool`；**战败必受伤 + `restUntilDay` 强制养伤一天**）/ `stay`（立即结算在家/院子/休息）。**包裹只被 travel/battle 消耗**：`startTravel`/`startBattle` 清空 `packedBag`；**`stayHome` 不动包裹**（在家不算用掉行李，bag 留在门口 + 宠物保持 `ready`，留给下次出门或满 24h 在首页过期；stay 的结算也不再读包裹内容，是一场独立的休息）。三种动作都各算「今天的一次行动」（都写 `lastActionDay`，当天不能再动）。「天」按 **UTC+8**（`dayKey`/`nextDayKey`）。battle 因要查 DB 匹配，逻辑在 day route 里，不在纯 `decideDay`。LLM 文本走 `src/lib/openrouter.ts` 的 `chatComplete/jsonComplete`（OpenRouter，默认模型 `deepseek/deepseek-v4-flash`；它是推理模型，客户端默认关推理 `reasoning.enabled=false`，否则正文会被 reasoning token 吃成 null）；对战判定封装在 `src/server/llm/battleJudge.ts`、旅行明信片封装在 `src/server/llm/postcard.ts`（都内置无 key 的程序化回退）。**明信片改在「归来时」生成**（旧 `travelStory.ts`/`Trip.llmPostcard` 是「出发时生成」——无 DB 列、出发即丢的死代码，已删；现在 `tickSave` 结算返程时才掷稀有度→定地标→喂 LLM 写，随存档落库）。图片理解仍走 `src/lib/minimax.ts`（`understandImage`，独立的 MiniMax 视觉端点）。旅行时长以小时计、用 `CAPY_TIME_SCALE` 在 dev 压缩。明信片 AI 生图仍下线（`generatePostcard` 只产文案，渲染程序化 `PostcardArt`）；相册新增「对战」Tab，对战结果复用 ResultScreen（`lastResult.kind==='battle'`）。
- **部署（Vercel）**：所有 route 文件均 `runtime="nodejs"`+`dynamic="force-dynamic"`。云存档需要 Supabase/Postgres，配置 `POSTGRES_URL`（优先 pooled）并依次运行迁移 `0001_storage_refactor.sql`、`0002_gameplay_v2.sql`（v2 加 `curiosity/rest_until_day/pending_stress*` 列 + `battle_pool` 表）。LLM 文本可选：`OPENROUTER_API_KEY`（默认模型 `deepseek/deepseek-v4-flash`，`OPENROUTER_MODEL`/`OPENROUTER_MAX_TOKENS`/`OPENROUTER_TIMEOUT_MS`/`OPENROUTER_REASONING` 可覆盖）；图片理解另用 `MINIMAX_API_KEY`（视觉端点 `MINIMAX_VLM_URL`）；`CAPY_TIME_SCALE`（dev 把旅行小时压成秒，默认 1）。不支持 Upstash/Vercel KV。`/agent/skill.md` 用点号文件夹路由，域名可用 `APP_BASE_URL` 固定。
- **UI 主题**：Tailwind v4 `@theme` in `globals.css`（`bg-cream`/`text-ink`/`bg-accent #D95F59`/`rounded-sticker`/`font-hand`）。标题字体用系统 CJK（Yuanti/PingFang），不外链 Google Fonts。`ErrorBoundary.tsx` 包住每屏，3D 出错只显示重试卡片不至于全白。
- **验证 & 坑**：dev 运行时用 `npx tsc --noEmit`，**不要在 `next dev` 运行时跑 `next build`**（两者共用 `.next`，会让 dev 加载不稳定/白屏；修复：`rm -rf .next` 后重启 dev）。
- 仓库目录名 `Capybanana` 含大写，npm 包名不能含大写——脚手架曾用小写临时子目录再上移。
- **远期玩法蓝图（用户已提出，部分实现）**：已落地——旅行→明信片闭环、对战→对战记录 + 负伤与强制休养、按 UTC+8「天」推进的日循环、好奇心成长倾向、Agent 压力上报。仍待做——纪念品/特产体系化、家庭活动、在家自主生活由 时间/心情/体力/是否受伤/纪念品 驱动选择活动、对战对手 rating 双向更新与好友对战。
